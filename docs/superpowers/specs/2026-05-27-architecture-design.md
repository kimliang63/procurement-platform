## 二、服务拆分

### 2.1 Web Server

| 项目 | 说明 |
|------|------|
| 职责 | 静态资源托管、前端 SPA、飞书 SSO 登录 |
| 技术栈 | React + Ant Design + Vite |
| 部署 | 飞书云托管 / Vercel / 自建 Nginx |
| 认证 | 飞书 OAuth2 → JWT Token |

### 2.2 Bot Server

| 项目 | 说明 |
|------|------|
| 职责 | 飞书消息接收、LLM 语义理解、卡片交互回调 |
| 技术栈 | Node.js (Express) + 飞书 SDK + LLM API |
| 部署 | 飞书云托管 / 自建服务器 |
| 事件 | im.message.receive_v1、card.action.trigger |

### 2.3 API Server

| 项目 | 说明 |
|------|------|
| 职责 | 业务逻辑、权限校验、数据校验 |
| 技术栈 | Node.js (Express / Koa) 或 Python (FastAPI) |
| 部署 | 与 Bot Server 同进程或独立部署 |
| 认证 | JWT Token 校验 |

### 2.4 MCP Server

| 项目 | 说明 |
|------|------|
| 职责 | 封装飞书多维表格 API，提供统一数据操作接口 |
| 技术栈 | Node.js / Python |
| 部署 | 与 API Server 同进程 |
| 核心 | 13 个 MCP 工具，Web 和 Bot 共用 |

### LLM Prompt 设计要点

```markdown
你是采购协同平台的 Bot 助手。用户会用自然语言描述操作需求。

## 你的能力（可调用的工具）
- create_project: 创建项目
- update_project: 更新项目
- delete_project: 删除项目
- get_project: 查询项目详情
- list_projects: 查询项目列表
- advance_node: 推进节点状态
- update_node: 更新节点信息
- mark_node_abnormal: 标记节点异常
- create_issue: 创建问题
- update_issue: 更新问题
- list_issues: 查询问题列表
- get_stats: 查询统计数据
- get_summary: 生成汇总

## 处理规则
1. 识别用户意图，选择最匹配的工具
2. 提取工具所需参数
3. 参数不完整时，追问用户补充
4. 项目名称模糊时，列出候选让用户确认
5. 删除等不可逆操作，要求二次确认
6. 无权限时，提示用户联系管理员

## 输出格式
返回 JSON:
{
  "intent": "工具名",
  "params": { ... },
  "confirm_required": false,
  "message": "追问/确认消息（如需要）"
}
```

## 五、数据流

### 5.1 Web 端操作流

```
用户操作 → React 组件 → API Server → MCP Server → 飞书多维表格 API → Bitable
                                                                    ↓
用户界面 ← React 状态更新 ← API 响应 ← MCP 返回 ← API 响应 ← 数据返回
```

### 5.2 Bot 对话流

```
用户发消息 → 飞书服务器 → Bot Server webhook
  ↓
解析消息 → LLM 语义理解 → 提取意图+参数
  ↓
MCP Server → 飞书多维表格 API → Bitable
  ↓
生成回复 → 飞书卡片/文本 → 用户
```

### 5.3 卡片交互流

```
用户点击卡片按钮 → 飞书服务器 → Bot Server webhook (card.action.trigger)
  ↓
解析 action → 调用 MCP 工具 → 更新 Bitable
  ↓
更新卡片内容 → 返回给飞书 → 用户看到更新后的卡片
```

## 六、部署架构

### 方案 A：飞书云托管（推荐 v1）

```
飞书云托管（Bot Server + API Server + MCP Server）
  ├── Node.js 应用
  ├── 飞书 SDK
  └── LLM API 调用

静态资源（Web 前端）
  ├── 飞书云托管静态资源
  └── 或 Vercel / CDN

数据（飞书多维表格）
  └── 无需部署，飞书托管
```

### 方案 B：自建服务器

```
Nginx（反向代理 + 静态资源）
  ├── /          → Web SPA
  ├── /api       → API Server
  └── /webhook   → Bot Server

API Server + Bot Server（同进程或独立）
  └── PM2 / Docker

数据库：飞书多维表格（无需自建）
```

## 七、关键技术决策

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 数据存储 | 飞书多维表格 | 零运维，Bot 直接读写 |
| Bot 框架 | 飞书 SDK + Webhook | 官方支持，稳定 |
| 语义理解 | LLM API | 灵活，不硬编码指令 |
| MCP 层 | 自建 MCP Server | 统一数据访问，Web/Bot 共用 |
| 前端框架 | React + Ant Design | 原型已基于 React |
| 认证 | 飞书 OAuth2 | 统一身份，SSO |
| 部署 | 飞书云托管（v1） | 最快上线，零运维 |
