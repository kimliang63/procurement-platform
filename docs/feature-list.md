## 二、飞书 Bot 功能（语义理解 + MCP 执行）

### 架构原则

```
用户自然语言输入 → LLM 语义理解 → 提取意图+参数 → 调用 MCP Tool → 执行结果 → 卡片回复
```

**核心设计**：不硬编码指令，由模型理解用户意图后调用对应 MCP 工具执行。

### 2.1 卡片消息能力

| # | 卡片类型 | 触发时机 | 推送对象 | 原型状态 |

### 2.2 对话式操作（语义理解）

用户通过自然语言与 Bot 对话，LLM 理解意图后调用 MCP 工具执行。

#### 项目操作

| # | 用户意图示例 | 语义理解要点 | 执行动作 | MCP 工具 |
|---|-------------|-------------|---------|----------|
| 2.2.1 | "帮我创建一个XX设备采购项目" | 提取：项目名称、品类 | 创建项目 | `create_project` |
| 2.2.2 | "XX设备采购的预算是500万" | 提取：项目名称、预算数值 | 更新项目预算 | `update_project` |
| 2.2.3 | "把ZZ服务采购的负责人改成李四" | 提取：项目名称、新负责人 | 更新负责人 | `update_project` |
| 2.2.4 | "XX设备采购定标完成了" | 提取：项目名称、节点名称 | 推进节点状态 | `advance_node` |
| 2.2.5 | "所有进行中的项目列一下" | 提取：筛选条件（状态=进行中） | 查询项目列表 | `list_projects` |
| 2.2.6 | "XX设备采购现在什么进度" | 提取：项目名称 | 查询项目详情 | `get_project` |
| 2.2.7 | "删除ZZ服务采购项目" | 提取：项目名称、确认意图 | 删除项目 | `delete_project` |

| 2.2.10 | "XX设备采购招标答疑的负责人改成王五" | 提取：项目名称、节点名称、新负责人 | 更新节点负责人 | `update_node` |

### 2.3 语义理解规则

| 规则 | 说明 |
|------|------|
| 项目名称模糊匹配 | "XX设备" 匹配 "XX设备采购"，多匹配时追问确认 |
| 节点名称同义词 | "发标"="发放标书"="发布招标"，需统一映射 |
| 意图不确定时追问 | "帮我处理一下XX" → 追问 "请问是要更新状态、添加问题，还是其他操作？" |
| 缺少必填参数时追问 | "创建一个采购项目" → 追问项目名称和品类 |
| 操作确认 | 删除等不可逆操作需二次确认 |
| 权限校验 | 操作前校验用户是否有对应项目的操作权限 |

### 2.4 MCP 工具清单

| 工具名 | 功能 | 参数 | 返回 |
|--------|------|------|------|
| `create_project` | 创建项目 | name, category, owner, budget, planStart, planEnd, remark | 项目详情 |
| `update_project` | 更新项目 | projectId, {name, owner, budget, planStart, planEnd, remark} | 项目详情 |
| `delete_project` | 删除项目 | projectId | 确认结果 |
| `get_project` | 查询项目详情 | projectId | 项目详情+节点列表 |
| `list_projects` | 查询项目列表 | {category, status, stage, owner} | 项目列表 |
| `advance_node` | 推进节点状态 | projectId, nodeKey, status | 节点详情 |
| `update_node` | 更新节点信息 | projectId, nodeKey, {assignee, planDate, note} | 节点详情 |
| `mark_node_abnormal` | 标记节点异常 | projectId, nodeKey, reason | 节点详情 |
| `create_issue` | 创建问题 | projectId, stage, desc, assignee, priority | 问题详情 |
| `update_issue` | 更新问题 | issueId, {status, priority, assignee, desc} | 问题详情 |
| `list_issues` | 查询问题列表 | {projectId, status, priority} | 问题列表 |
| `get_stats` | 查询统计数据 | {dateRange, category, stage} | 统计数据 |
| `get_summary` | 生成汇总 | {dateRange, type} | 汇总内容 |

### 2.5 对话流程示例

```
用户: 帮我创建一个新项目，名称是CC服务采购，品类是服务，负责人是赵六
  ↓
LLM 理解意图: create_project
  ↓
LLM 提取参数: {name: "CC服务采购", category: "服务", owner: "赵六"}
  ↓
缺少必填: budget, planStart, planEnd
  ↓
Bot 回复: 好的，创建项目「CC服务采购」，还需要以下信息：
         1. 预算金额（万元）
         2. 计划开始日期
         3. 计划结束日期
  ↓
用户: 预算100万，6月1号到8月31号
  ↓
LLM 补全参数: {budget: 100, planStart: "2026-06-01", planEnd: "2026-08-31"}
  ↓
调用 MCP: create_project({name: "CC服务采购", category: "服务", owner: "赵六", budget: 100, ...})
  ↓
Bot 回复卡片: 项目创建成功 ✅
  项目名称: CC服务采购
  编号: CG-2026-007
  负责人: 赵六
  当前阶段: 需求确认
```

## 五、数据模型（飞书多维表格）

### 5.1 项目表（projects）

| 字段名 | 字段类型 | 说明 |
|--------|---------|------|
| project_id | 自动编号 | 主键 |
| name | 文本 | 项目名称，如「XX设备采购」 |
| no | 文本 | 项目编号，如 CG-2026-001 |
| category | 单选 | 采购品类：原材料 / 设备 / 服务 |
| owner | 人员 | 项目负责人 |
| budget | 数字 | 预算金额（万元） |
| plan_start | 日期 | 计划开始日期 |
| plan_end | 日期 | 计划结束日期 |
| current_stage | 单选 | 当前阶段（13个节点 key） |
| status | 单选 | 项目状态：正常 / 异常 / 已完成 |
| risk | 文本 | 风险描述 |
| remark | 多行文本 | 备注 |
| suppliers | 多行文本 | 供应商列表（JSON数组） |
| created_at | 创建时间 | — |
| updated_at | 更新时间 | — |

### 5.2 节点表（nodes）

| 字段名 | 字段类型 | 说明 |
|--------|---------|------|
| node_id | 自动编号 | 主键 |
| project_id | 关联项目表 | 所属项目 |
| stage_key | 单选 | 阶段标识（requirement / supplier_dev / ...） |
| stage_label | 文本 | 阶段中文名 |
| order | 数字 | 阶段顺序（1-13） |
| status | 单选 | 节点状态：pending / in_progress / completed / blocked |
| assignee | 人员 | 节点负责人 |
| plan_date | 日期 | 计划完成日期 |
| actual_date | 日期 | 实际完成日期 |
| note | 多行文本 | 备注 |
| abnormal_reason | 多行文本 | 异常原因（异常时填写） |

### 5.3 问题表（issues）

| 字段名 | 字段类型 | 说明 |
|--------|---------|------|
| issue_id | 自动编号 | 主键 |
| project_id | 关联项目表 | 所属项目 |
| stage_key | 单选 | 关联阶段 |
| description | 多行文本 | 问题描述 |
| assignee | 人员 | 责任人 |
| priority | 单选 | 优先级：high / medium / low |
| status | 单选 | 状态：open / in_progress / closed |
| created_at | 创建时间 | — |
| updated_at | 更新时间 | — |

### 5.4 用户表（users）

| 字段名 | 字段类型 | 说明 |
|--------|---------|------|
| user_id | 自动编号 | 主键 |
| feishu_open_id | 文本 | 飞书 open_id |
| feishu_user_id | 文本 | 飞书 user_id |
| name | 文本 | 用户姓名 |
| avatar | 文本 | 头像 URL |
| role | 单选 | 角色：admin / pm / member |
| created_at | 创建时间 | — |

### 5.5 表间关系

```
projects (1) ──< (N) nodes
projects (1) ──< (N) issues
users (1) ──< (N) projects    （owner）
users (1) ──< (N) nodes       （assignee）
users (1) ──< (N) issues      （assignee）
```
