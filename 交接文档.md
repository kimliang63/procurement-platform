## 一、项目概览

| 项 | 说明 |
|---|------|
| 项目名称 | 供应链采购协同平台（Procurement Platform） |
| 生产地址 | **https://hras-ai-asrm.ztn.cn/**（前端 + 后端同域） |
| 代码仓库 | https://github.com/kimliang63/procurement-platform |
| 主分支 | `main` |
| 技术栈 | React 18 + Ant Design 5 + Tailwind CSS 4 + Express 4 + MySQL |
| 部署方式 | Docker 容器运行在内网服务器，HRAS 壳子 nginx 反向代理到公网 |
| 数据库 | MySQL 8.0（`assc_srm`），内网 10.100.12.119:63307 |
| Bot | 飞书机器人「采购项目小助手」 |

### 15 阶段采购流程

需求确认 → 供应商开发 → 技术交流 → 打样 → 招标方案审批 → 发标 → 答疑 → 供应商回标 → 开标 → 定标 → 中标/未中标通知 → 合同审批 → 生产 → 运输 → 验收

### 核心功能模块

| 模块 | 说明 |
|------|------|
| Dashboard 项目看板 | 统计卡片 + BU/负责人图表 + 任务类型饼图 |
| 项目列表 | 15 节点时间线视图，节点状态颜色标记 |
| 项目详情 | 节点表格 + 问题列表，支持推进/异常标记 |
| 问题追踪 | 项目问题 CRUD，按优先级/状态筛选 |
| 用户管理 | 角色管理（admin/pm），仅管理员可见 |
| 飞书 Bot | 群聊绑定、自然语言交互（DeepSeek LLM）、周报推送 |
| 业务规则引擎 | 三维矩阵（采购方式 × 预算 × 是否单一来源）决定节点可见性 |

### 2.2 关键设计决策

1. **MCP 层是共享内核**：REST API 和飞书 Bot 都调用同一套 MCP 函数，保证业务逻辑一致
2. **`db.js` 兼容层**：API 签名与旧 Bitable 层一致（`listRecords`, `getRecord`, `createRecord` 等），迁移时无需改 MCP 和 routes
3. **Bot 异步处理**：Webhook 收到消息后立即返回 200，业务逻辑 fire-and-forget 执行（8 秒超时兜底）
4. **Webhook 去重**：`processedEvents` Set 按 event_id 去重，5 分钟自动清理
5. **前后端同源**：Docker 镜像包含前端构建产物，Express 同时托管前端页面和 API，使用相对路径 `/api`，无需跨域配置

### 2.3 权限模型

| 角色 | 权限 |
|------|------|
| admin（管理员） | 查看所有项目 + 访问用户管理页面 |
| pm（项目负责人） | 仅查看自己负责的项目 |

- 后端：`extractUser` 中间件解析 Bearer token → `filterByOwner` 按角色过滤
- 前端：`AdminRoute` 组件检查 `user.role === 'admin'`

### 2.4 业务规则引擎

节点可见性由三维矩阵决定（`server/src/mcp/rules.js`）：
- `isSingleSource`（是/否）
- `budget`（<100万 / ≥100万）
- `procurementMethod`（框架类 / 项目类）

每个节点取值：`required`（必填）/ `visible`（可选）/ `hidden`（隐藏）。
快速规则（<100万/单一来源）仅 4 个必填节点，招标规则（≥100万/框架招标）12 个必填节点。

```

    │
```

## 四、数据库结构

数据库 `assc_srm` 运行在 MySQL 8.0，建表语句见 `server/src/db-schema.sql`。

| 表名 | 说明 | 关键字段 |
|------|------|---------|

| `projects` | 项目表 | name, no(自动编号), owner, budget, task_type, status |

### 5.2 第一步：获取代码

```bash

git clone https://github.com/<你的GitHub用户名>/procurement-platform.git
cd procurement-platform
```

### 5.3 第二步：接管飞书应用

现有飞书应用「采购项目小助手」（App ID: `cli_aa9c0ff0f8b99cc0`）由原管理员转让给你，无需重新创建。

1. 联系原管理员（梁景悦）在 [飞书开放平台](https://open.feishu.cn) → 应用管理 → 成员管理 中将你添加为**管理员**
2. 确认以下权限已开启（已配置好，无需改动）：
   - `im:message:send_as_bot` — 发送消息
   - `im:chat:readonly` — 读取群聊信息
   - `contact:user.base:readonly` — 读取用户基本信息
   - `authen:user.info:read` — OAuth 登录
3. 确认事件订阅 Webhook 地址为：`https://hras-ai-asrm.ztn.cn/webhook/bot`
4. 确认 OAuth 重定向 URL 包含：`https://hras-ai-asrm.ztn.cn/auth/callback`

> **注意**：App ID 和 App Secret 保持不变，环境变量中的 `FEISHU_APP_ID` / `FEISHU_APP_SECRET` 沿用现有值。

### 5.4 第三步：获取 DeepSeek API Key

1. 登录 [DeepSeek 开放平台](https://platform.deepseek.com)
2. 创建 API Key
3. 确保账户有余额（用于 LLM 调用）

### 5.5 第四步：准备 MySQL 数据库

**选项 A：使用现有数据库**
- 联系管理员获取 `10.100.12.119:63307` 的访问权限
- 数据库 `assc_srm` 已建好表结构

**选项 B：自建 MySQL**
```bash
docker run -d --name mysql \
  -e MYSQL_ROOT_PASSWORD=<你的密码> \
  -e MYSQL_DATABASE=assc_srm \
  -p 3306:3306 \
  mysql:8.0

mysql -h 127.0.0.1 -u root -p assc_srm < server/src/db-schema.sql
```

### 5.6 第五步：配置环境变量

```bash
cp .env.docker.example .env.docker
```

编辑 `.env.docker`：

## 八、常见问题排查

### 8.1 Bot 不回复消息

1. 检查服务是否运行：`curl https://hras-ai-asrm.ztn.cn/health`
2. 检查飞书开放平台 → 事件订阅 → Request URL 是否为 `https://hras-ai-asrm.ztn.cn/webhook/bot`
3. 检查 DeepSeek API Key 是否有效
4. 查看日志：`docker compose logs -f | grep -E "Error|Webhook|LLM"`

### 8.2 前端无法登录

1. 检查飞书应用 OAuth 重定向 URL 是否包含 `https://hras-ai-asrm.ztn.cn/auth/callback`
2. 检查 `CORS_ORIGIN` 是否为 `https://hras-ai-asrm.ztn.cn`

### 8.3 节点无法推进

1. 确认节点已填写「计划完成日期」
2. 确认上一个节点已完成（节点按顺序推进）
3. 检查业务规则引擎是否要求该节点必填
